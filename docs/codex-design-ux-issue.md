# Issue: デザイン・UX・図表レイアウトの改善

対象: Codex
種別: デザイン改善 / UX改善 / レイアウト再設計
優先度: 高

---

## 0. 結論（最初に読む）

コンテンツの中身（表・本文）は前回改善で良くなった。今回は**見せ方（レイアウト・UI・採点UX）**の問題。
大きく3つ:

1. **非インタラクティブな「図表」を、操作前提のワークベンチに押し込んで表が小さく潰れている。**
   → 操作できないものは「図表ワークベンチ」をやめ、本文中の正規コンテンツ（HTMLの表など）として全幅で見せる。
2. **ワイド液晶に最適化されていない。** コンテンツ幅が `min(1180px, 100%)` で固定され、横が余る。
3. **確認問題の採点が分かりにくい。** どの選択肢が正解で、自分がどこを間違えたかが視覚的に分からない。

> 修正対象は主に [`assets/css/styles.css`](../assets/css/styles.css)、[`assets/js/lesson.js`](../assets/js/lesson.js)、[`scripts/generate-site.js`](../scripts/generate-site.js)。
> HTMLは生成物なので、必ず `node scripts/generate-site.js` で再生成する。

---

## 1. 【構造】操作できない「図表」をワークベンチから外す（最重要）

### 現状の問題

図表種別ごとの「操作可否」と現状ページ数:

| 種別 | 操作 | ページ数 | 現状の見せ方 |
|---|---|---|---|
| distribution / regression / classification / attention | **可** | 18 | スライダー + Canvas/SVG（妥当） |
| comparison-table | 不可 | 21 | SVGの表をワークベンチ内に表示 |
| relation-table | 不可 | 19 | 同上 |
| flow | 不可 | 27 | SVGのフロー図 |
| network | 不可 | 2 | SVGの構造図 |

→ **87ページ中69ページ(約79%)は操作できない。**

それなのに全ページが同じ `.visual-workbench`（[styles.css:413](../assets/css/styles.css#L413)）= 「コントロール列(17rem) + 出力(1fr)」の2カラム構成。
操作できないページでは:

- 左の17rem列に「このテーマは数値を操作せず…」という**注記だけ**が入り、横幅を無駄食い（[lesson.js:82-84](../assets/js/lesson.js#L82)）。
- 残り幅にSVGの表が `viewBox 900x430` で縮小表示され、**文字が小さく読みにくい**。
- そもそも「インタラクティブ図表」という見出しなのに動かない。看板倒れ。

### あるべき姿（この方針で実装すること）

**「操作できないものは図表ワークベンチをやめる」。** 種別ごとに描画コンテナを分岐する。

- **comparison-table / relation-table → 本物のHTML `<table>`** を本文フロー内に**全幅**で描画する。
  - SVGの表をやめる。`<section class="lesson-section">` の通常コンテンツとして、見出しを「インタラクティブ図表」ではなく内容に即した見出し（例「論点の整理」「比較表」）にする。
  - コントロール列(17rem)は**出さない**。
  - スタイル: ストライプ・ヘッダ色・十分な行高・折返し許可。下記2-3のCSS参照。
- **flow / network → SVGのまま**でよいが、**コントロール列を出さず全幅**にする。見出しも「図表」より「処理の流れ」「層構造」等に。
- **distribution / regression / classification / attention → 現状のワークベンチを維持**（操作できるので2カラムでよい）。ただし2章のワイド最適化は適用。

### 実装の当て所

- [`scripts/generate-site.js`](../scripts/generate-site.js) の `lessonPage()`（[L338付近](../scripts/generate-site.js#L338)）で、`item.visual.type` を見て
  - 操作可能型 → 現状の `.visual-workbench`（canvas/svg + controls）を出力
  - 表型 → サーバ側でHTML `<table>` を直接生成して本文に埋め込む（JS描画に頼らない＝SEO/初期表示にも有利）
  - flow/network → controls無しのSVGコンテナ
- [`assets/js/lesson.js`](../assets/js/lesson.js) の `buildControls()` で表型に注記を出す処理（[L82-84](../assets/js/lesson.js#L82)）は不要になるので削除。

## 2. 【デザイン】ワイド液晶に最適化する

### 現状の問題

- 主要コンテナがすべて `width: min(1180px, 100%)` 固定（[styles.css:232](../assets/css/styles.css#L232), [L595](../assets/css/styles.css#L595)）。
  → 1920px / 2560px のワイド液晶で左右に広い余白ができ、間延びして見える。
- `main` の左右パディングは `clamp(1rem, 4vw, 4rem)`（[L223](../assets/css/styles.css#L223)）で上限4rem。
- Canvasは `aspect-ratio: 900/500` 固定で、広い画面でも大きくならない（[L694-702](../assets/css/styles.css#L694)）。

### あるべき姿

1. **コンテンツ最大幅を広げる。** 例: 本文系は `min(1480px, 100%)`、レッスン本文の可読幅は別途確保。
   - 読み物（本文段落）は可読性のため1行を広げすぎない（目安: 70〜80全角）。**「コンテナは広く、本文段落は読みやすい幅に制限」**という二段構えにする。
   - 推奨: CSS変数で `--content-max: 1480px;`（または `clamp`）を定義し、各 `.layout/.lesson-shell/...` で使う。`@media (min-width: 1800px)` で `1640px` 程度までさらに拡張してもよい。
2. **レッスンページを3カラム化（ワイド時）。** 現状 `.lesson-layout` は「本文(1fr) + サイド(18rem)」の2カラム（[L643](../assets/css/styles.css#L643)）。
   - ワイド時は「目次/用語サイド(左) + 本文(中央) + 進捗/前後ナビ(右)」のように横を活用する案を検討。最低でも本文カラムをもっと広く使う。
3. **操作可能な図表ワークベンチを大きく。** ワイド時はCanvas/SVGの表示領域を拡大（`aspect-ratio` 維持のまま最大幅を上げる、またはコントロール列を固定幅にして出力を可変最大化）。
4. **余白・タイポの調整。** セクション間 `gap`、`.lesson-main` の `padding`（現状1.25rem）をワイド時に拡大。情報密度が低く見えないよう、カードのグリッド列数もワイドで増やす（`minmax` の下限調整）。
5. ブレークポイント追加: 現状 900px / 560px のみ（[L863](../assets/css/styles.css#L863), [L884](../assets/css/styles.css#L884)）。`min-width: 1280px` / `1800px` の拡張用ブレークポイントを足す。

> 数値（1480/1640等）は目安。最終的に1920・2560で間延びせず、ノートPC(1366/1440)でも破綻しないことを確認すること。

## 3. 【機能/UX】確認問題の採点を分かりやすくする（最重要）

### 現状の問題

[`lesson.js` の `setupQuiz()`](../assets/js/lesson.js#L106-117) は採点時に:

- 正解数と「正解:/確認: 解説文」を**1つのテキスト塊**として `quizResult` に出すだけ。
- **選択肢自体の色やマークが変わらない**（どれが正解か、自分がどこを選んで間違えたかが分からない）。
- 正解の選択肢テキストを明示しない。「確認: <解説>」だけ出ても、結局どれが正解か探す羽目になる。

一方、4択演習ページ([`practice.js`](../assets/js/practice.js))は `is-correct` / `is-wrong` で選択肢を色分けする良いUIを持っている（CSS [styles.css:817-849](../assets/css/styles.css#L817)）。**レッスンの確認問題だけ体験が劣っている。**

### あるべき姿

採点ボタン押下時に、**問題ごと・選択肢ごとに正誤を視覚化**する。practiceページのパターンを流用する。

- 正解の選択肢に `is-correct`（緑＋✓）を付ける。
- ユーザーが選んだ選択肢が不正解なら `is-wrong`（赤＋✗）を付ける。
- 問題カード単位で「正解 / 不正解」のラベルと解説を、その問題の直下に表示（全部まとめた塊ではなく）。
- 未回答の問題は「未回答」と明示。
- 採点後はラジオを `disabled` にする、または「もう一度」で再挑戦できるようにする。
- 既存の `.quiz-option` / `.quiz-card` に practice と同等の正誤スタイルを追加（CSSを共通化できるとなお良い）。

### 実装の当て所

- [`assets/js/lesson.js`](../assets/js/lesson.js) `setupQuiz()` を、選択肢DOMにクラスを付与する方式へ書き換え。
- [`assets/css/styles.css`](../assets/css/styles.css) に `.quiz-option.is-correct` / `.quiz-option.is-wrong`（practiceの `.practice-option` 同等）を追加。
- 正解選択肢の判定は `item.answer` と各 `<input value>` の一致で可能（既存データで足りる）。

## 4. 【UX】その他の改善（中〜小）

- **見出し語の整合**: 操作できないページの「インタラクティブ図表」見出しは誤解を招く。内容に応じた見出しへ（1章の方針に合わせる）。
- **左サイド(`.lesson-side`)の活用**: 現状「このページの用語」＋前後リンクのみ。ワイド化に合わせて、章内の他レッスンへのミニ目次や進捗表示を入れると回遊性が上がる。
- **進捗の可視化**: 「完了済み」テキストだけでなく、章インデックスやトップで完了状態が一目で分かるようにする（`localStorage` の `gken-lesson-progress-v3` を利用）。
- **トップ/章インデックスのカード密度**: ワイド時にカード列数を増やし、スカスカに見えないように。
- **ダークモードでの表**: 新設するHTML表が両テーマで破綻しないこと（CSS変数で色指定）。

## 4.5. 【UX】採点エリアに「次のページへ」ボタンを追加

### 目的

学習→採点まで終えたら、その場で次のレッスンへ進めるようにして回遊効率を上げる。
現状、次ページへの導線は左サイド `.lesson-side` の「前後の学習」リンクのみ（[generate-site.js:363-365](../scripts/generate-site.js#L363)）で、採点後に視線を大きく移す必要がある。

### あるべき姿

採点ボタンと同じ `.quiz-actions`（[generate-site.js:347-349](../scripts/generate-site.js#L347)）に、**「次のページへ」ボタン**を追加する。

- 次レッスンのURLは生成スコープで既に算出済み（`next` 変数 [generate-site.js:306](../scripts/generate-site.js#L306)）。それを `href` にしたリンクボタンを置く。
- ボタンは `<a class="primary-button">` 等で、`採点する` `このページを完了` の並びに配置（推奨は右寄せ or 末尾）。
- **最終ページ（`next` が無い）では非表示**にする（または「章のトップへ」「学習マップへ」に切替）。
- 望ましい挙動: 採点 or 「このページを完了」後に強調表示（例: 採点までは控えめ、完了後に `primary-button` で目立たせる）。最低限、常時クリック可能でもよい。
- 既存の左サイド「前後の学習」リンクは**残す**（重複してよい。サイドは前後両方、採点エリアは「次へ」中心）。
- 「前のページへ」も併記すると親切（任意）。

### 実装の当て所

- [`scripts/generate-site.js`](../scripts/generate-site.js) の `lessonPage()` の `.quiz-actions` 出力に、`next ? '<a class="primary-button" href="...">次のページへ</a>' : '<a class="ghost-button" href="./index.html">章のトップへ</a>'` 相当を追加。
- 「完了後に強調」を入れる場合のみ [`assets/js/lesson.js`](../assets/js/lesson.js) の `setupProgress()` / `setupQuiz()` でクラス切替。

### 受け入れ基準

- [ ] 採点エリアから次レッスンへ1クリックで遷移できる。
- [ ] 最終ページでは「次へ」が出ず、章トップ/学習マップ等へ誘導される。
- [ ] 既存の左サイドの前後リンクは維持されている。

## 5. 受け入れ基準（Acceptance Criteria）

### 構造（1章）
- [ ] comparison-table / relation-table のページが、**HTMLの `<table>`** で全幅表示されている（SVGの縮小表ではない）。
- [ ] 操作できないページに、コントロール列（17rem）や「数値を操作せず…」の注記が**残っていない**。
- [ ] 操作できるページ（distribution/regression/classification/attention）はワークベンチUIを維持している。

### デザイン（2章）
- [ ] 1920px・2560px幅で、コンテンツが間延びせず横幅を活用している。
- [ ] 本文段落の1行が長すぎず読みやすい幅に保たれている。
- [ ] 1366px / 1440px のノートPCでレイアウトが破綻しない。
- [ ] 操作可能な図表が、ワイド画面で十分大きく表示される。

### 採点UX（3章）
- [ ] 確認問題を採点すると、**正解の選択肢が緑、誤って選んだ選択肢が赤**で明示される。
- [ ] 各問題の直下に正誤と解説が表示される（全問まとめた1塊ではない）。
- [ ] 未回答の扱いが明示される。
- [ ] レッスン確認問題と4択演習で、正誤表示の体験が概ね揃っている。

### 共通
- [ ] すべて `node scripts/generate-site.js` 実行後にHTMLへ反映される（HTML直編集ではない）。
- [ ] ライト/ダーク両テーマで破綻しない。

## 6. 推奨着手順

1. **3章（採点UX）** — 影響範囲が `lesson.js` + CSS少量で、体感改善が最も大きい。
2. **1章（表の全幅HTML化）** — 69ページの見え方が一気に改善。
3. **2章（ワイド最適化）** — CSS中心。1章で表が全幅になった後にやると効果を確認しやすい。

## 7. やってはいけないこと

- 操作できない内容を「インタラクティブ図表」ワークベンチのまま放置する。
- 表を縮小SVGで描き続ける（読みにくい）。
- コンテナだけ広げて本文段落まで横長にし、可読性を下げる。
- 採点結果をテキスト1塊で返すだけにとどめる。
- HTMLを直接編集して済ませる（再生成で消える）。
