# Issue: 学習ページ本文の充実（全87ページ著作）

対象: Codex
種別: コンテンツ拡充（教材本文）
優先度: 高

---

## 0. 結論・方針

各ページの「本文解説」が薄い（**実測: 中央値221字 / 3段落 / 最短128字**）。学習教材として不足。

方針: **全87レッスンの本文を、決まった型に沿って人手品質で著作する**（テンプレ自動生成からの脱却）。
現状 `content/lesson-content.json` の `explanation` は **69件中0件しか使われていない**＝本文はテンプレ依存。これを全廃し、著作本文へ置き換える。

> ⚠ これは「文章の水増し」ではない。**構造化して中身を増やす**。冗長な定型文を増やすのは禁止。
> 質 > 文字数。1ページ目安 **400〜800字**＋下記の型。

実装対象: [`content/lesson-content.json`](../content/lesson-content.json)（本文データ）、[`scripts/generate-site.js`](../scripts/generate-site.js)（描画）、[`assets/css/styles.css`](../assets/css/styles.css)（小見出し/コールアウト）。
HTMLは生成物。`node scripts/generate-site.js` で再生成する。

---

## 1. 本文の「型」（全ページ共通の骨格）

各レッスンを次の観点で構成する（不要な項目は省略可、ただし最低でも定義・仕組み・試験観点は必須）:

1. **定義** — 用語を一文で正確に。
2. **位置づけ / なぜ重要か** — 他概念との関係、登場した背景。
3. **仕組み・考え方** — どう動くか / どう使うか。手順や条件があれば箇条書き。
4. **具体例** — 試験でイメージしやすい身近な例。
5. **試験で問われる観点・ひっかけ** — 頻出論点、間違えやすいポイント（コールアウトで強調）。
6. **関連用語との区別** — `terms` 同士の境界（混同しやすい語の違い）。

## 2. データ構造

`lesson-content.json` の各レッスンに、構造化した本文を持たせる。
**ブロック配列**にして、見出し・段落・箇条書き・コールアウトを表現できるようにする。

```jsonc
{
  "confusion-matrix": {
    "visual": { "...": "既存のまま" },
    "body": [
      { "h": "定義", "p": "混同行列は、分類モデルの予測を実際のクラスと突き合わせ、TP・FP・FN・TN の4区分で集計した表。" },
      { "h": "なぜ重要か", "p": "正解率だけでは見えない誤りの偏りを把握でき、適合率・再現率・F値などの評価指標がここから導かれる。" },
      { "h": "仕組み", "list": [
        "TP: 陽性を正しく陽性と予測",
        "FP: 陰性を誤って陽性と予測（偽陽性）",
        "FN: 陽性を見逃す（偽陰性）",
        "TN: 陰性を正しく陰性と予測"
      ]},
      { "h": "具体例", "p": "病気の検査で、見逃し(FN)を減らしたいなら再現率を、誤検知(FP)を抑えたいなら適合率を重視する、といった使い分けに直結する。" },
      { "callout": "exam", "text": "FP と FN の取り違えは頻出のひっかけ。『偽陽性=陰性なのに陽性と判定』を必ず押さえる。" },
      { "h": "関連用語の区別", "table": [
        ["適合率", "陽性と予測したうち実際に陽性だった割合 = TP/(TP+FP)"],
        ["再現率", "実際の陽性のうち拾えた割合 = TP/(TP+FN)"]
      ]}
    ]
  }
}
```

ブロック種別:
- `{ "h": 見出し, "p": 段落 }` または `{ "h":…, "p":[…,…] }`（複数段落）
- `{ "h":…, "list": [..] }` 箇条書き
- `{ "callout": "exam"|"point"|"warn"|"example", "text": "…" }` コールアウト（[配色issue](./codex-visual-polish-issue.md)のコールアウトと共通）
- `{ "h":…, "table": [[左,右], …] }` 区別表（任意）

> 既存の `explanation`（未使用）から `body` へ移行。命名は実装者判断でよいが、**1つに統一**する。

### カバレッジの注意
- `lesson-content.json` は現状 **69件**（表/フロー系）。数値系18レッスン（regression, classification, distribution, neural-network 等）は未登録。
- **本文は全87レッスンに必要**。18件分のエントリを追加し、`body` を著作する（`visual` は計算系なので不要）。

## 3. 生成側の対応

[`scripts/generate-site.js`](../scripts/generate-site.js):

- `body` があれば**それを描画**（見出し`<h4>`/段落/箇条書き/コールアウト/区別表）。テンプレの `explanationFor()` は **`body` 未登録時のみのフォールバック**に降格。
- 既存テンプレ文（下記）が著作済みページに**残らない**こと。
- [配色issue](./codex-visual-polish-issue.md)の「キーワード強調」「コールアウト」スタイルと連携。

撲滅対象のテンプレ文（フィンガープリント）:
- 「〜では、…を別々の暗記事項ではなく、問題文の条件から使い分ける知識として扱います。」
- 「演習データでは「…」のように問われます。ポイントは、…」
- 「試験では、…の定義だけでなく、関連語との違いと利用場面が問われます。」

## 4. 品質基準（最重要）

- **公式本文・公式例題の転載をしない**（READMEの方針）。自分の言葉で説明する。
- **事実の正確性を担保する**。誤った定義は教材として致命的。著作後に**内容レビューを必ず1回かける**（章ごとに見直す）。
- **ページごとに内容が明確に異なる**こと。用語名だけ差し替えた同型文は不可。
- 1段落を長くしすぎない。小見出しで区切り、スキャンしやすく。
- 強調（`strong`/`mark`）は1段落1〜2か所まで（[配色issue](./codex-visual-polish-issue.md)準拠）。

検証用grep（0件であるべき）:
```bash
grep -rl "別々の暗記事項ではなく" chapters
grep -rl "演習データでは" chapters
grep -rl "関連語との違いと利用場面が問われます" chapters
```

## 6. 進め方（87ページは多い）

- **章単位**で進める（1章ぶん著作→事実レビュー→次章）。各章のチェックリストを作る。
- **先に“お手本”を1ページ完全著作**して品質ラインを固定（上記 `confusion-matrix` の例を基準にしてよい）。それを承認してから残りへ展開する。
- 大量生成時も、各ページを必ず**個別に**作る（テンプレ展開で量産しない）。

---

## 7. やってはいけないこと

- 定型文の水増しで字数だけ増やす。
- 用語名だけ差し替えた同型本文の量産。
- 公式資料の転載。
- 事実確認をしないまま大量生成。
- HTML直編集（再生成で消える）。
