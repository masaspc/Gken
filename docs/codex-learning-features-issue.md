# Issue: 学習支援機能の追加（模試・間隔反復・ダッシュボード ほか）

対象: Codex
種別: 機能追加（学習支援）
優先度: 高

---

## 0. 結論・スコープ

教材・図表・採点UIは整った。次は「合格支援ツール」にするための学習機能を**すべて**追加する。

実装する機能:

1. **共通基盤** … 問題IDの付与 + 進捗ストア共通化（他機能の前提）
2. **F1 模試モード（本番形式）** … web本番に合わせ **100分** 制限・全カテゴリ横断・採点後にカテゴリ別分析
3. **F2 間隔反復による誤答復習（SRS）** … 誤答を日をまたいで再出題
4. **F3 学習ダッシュボード（弱点分析）**
5. **F4 サイト内検索**
6. **F5 問題のブックマーク/フラグ**
7. **F6 問題⇄レッスンの相互リンク**
8. **F7 進捗のエクスポート/インポート**
9. **F8 試験日からの学習プラン**

> 静的サイト前提。サーバなし・`localStorage` で完結させる。
> HTML/ページは [`scripts/generate-site.js`](../scripts/generate-site.js) の生成物。必ず `node scripts/generate-site.js` で再生成する。

## 2. 共通基盤（最優先・他機能の前提）

### 2-1. 問題への安定ID付与

SRS・ブックマーク・ダッシュボードは「問題を一意に識別」する必要がある。現状 `id` が無い。

- [`scripts/extract-gtest-data.js`](../scripts/extract-gtest-data.js) で各問題に **内容ベースの安定ID** を付与する（並び替えでも変わらないよう、`q` 本文＋`c` から短いハッシュを生成。例 `q-<8桁hex>`）。
- 生成後の `content/gtest-questions.json` の各要素に `id` を持たせる。
- 実行時フォールバック: 万一 `id` 欠落時は `q` 本文のハッシュで補う共通関数を `assets/js/` に用意。
- レッスン確認問題側のID（`<slug>-practice-<n>` 等）とは名前空間を分け、衝突させない。

### 2-2. 進捗ストアの共通化

各機能がバラバラに `localStorage` を触らないよう、**共通モジュール** `assets/js/store.js` を作る。

責務:
- `gken-*` キーの読み書きを集約（JSON parse/stringify、try/catch、バージョン管理）。
- 後述の各ストアへの get/set/update API を提供。
- 破損データ時は安全に初期化。

`localStorage` キー設計（バージョン付き）:

| キー | 用途 | 形 |
|---|---|---|
| `gken-lesson-progress-v3` | レッスン完了（既存） | `{ [lessonSlug]: true }` |
| `gken-srs-v1` | 間隔反復 | `{ [qid]: { box:number, due:number(ts), seen:number, correct:number, wrong:number, last:number(ts) } }` |
| `gken-bookmarks-v1` | フラグ | `{ [qid]: true }` |
| `gken-stats-v1` | 学習統計 | `{ days:{ [YYYY-MM-DD]: {answered,correct} }, byCat:{ [cat]:{answered,correct} } }` |
| `gken-exams-v1` | 模試履歴 | `[{ date:ts, total, correct, durationSec, byCat:{[cat]:{total,correct}} }]` |
| `gken-plan-v1` | 学習プラン | `{ examDate: "YYYY-MM-DD" }` |
| `gken-theme` | テーマ（既存） | `"light"|"dark"` |

> 既存 `gken-progress-v1` が未使用なら整理する（使用箇所を確認の上）。

## 3. 機能要件

### F1. 模試モード（本番形式）

**目的**: 本番（web・**100分**）に近い条件で通し演習し、合否感覚とカテゴリ別弱点をつかむ。

**要件**:
- 専用ページ `mock/index.html` を新設（生成スクリプトに追加）。トップ/ナビから導線。
- **制限時間100分のカウントダウンタイマー**を常時表示。残り時間で色を変える（例: 残り10分で警告色）。0で**自動提出**。
- 出題: 全9カテゴリから横断出題。問題数は設定可能な定数 `MOCK_QUESTION_COUNT`（**初期値の目安160**。公式の最新問題数に合わせて1箇所で変更できるようにする）。
  - カテゴリ別問題数に比例して配分し、偏らないようにする。
  - 出題・選択肢ともシャッフル（practice.js の方式を流用）。
- **本番同様、解答中は正誤・解説を出さない**。
- ナビゲーション: 前後移動、**未回答/フラグの一覧（問題パレット）**で任意問題へジャンプ、「後で見直す」フラグ。
- 「提出」ボタン（残り時間内）＋確認ダイアログ。提出 or 時間切れで採点。
- **結果画面**:
  - 総合スコア（％・正答数/総数）、所要時間。
  - **カテゴリ別正答率**（棒や表で弱点が一目で分かる）。
  - 合否の目安表示（注記として「目安。公式合格基準とは異なる」と明記）。
  - 誤答を **F2(SRS)** に投入、`gken-stats-v1` / `gken-exams-v1` を更新。
  - 「間違いだけ復習」「もう一度模試」導線。
- 中断対策: 進行中状態を一時保存し、リロードで復帰できると望ましい（任意）。

**受け入れ基準**:
- [ ] タイマーが100分から減り、0で自動提出される。
- [ ] 全9カテゴリから出題され、結果でカテゴリ別正答率が出る。
- [ ] 解答中に正解・解説が表示されない。
- [ ] 問題パレットで未回答・フラグが識別でき、任意問題へ移動できる。
- [ ] 結果が `gken-exams-v1` に履歴として残り、誤答がSRSに入る。

### F2. 間隔反復による誤答復習（SRS）

**目的**: 誤答を忘れる前に再出題し、定着させる。G検定は範囲が広く反復が要。

**要件**:
- Leitnerボックス方式（例: box 0〜5）。正解で昇格＝次回出題を先延ばし、誤答で box を下げ＝近い日に再出題。`due`（次回出題日時）を持つ。
- **採点が発生する全モード（レッスン確認問題・F1模試・通常演習・復習モード）から共通APIでSRSを更新**する。
- 復習導線:
  - 専用「今日の復習」ページ（または既存 `review/index.html` を機能拡張）で、**due を過ぎた問題のみ**を出題。
  - due が無い未学習問題も「新規」として少量混ぜられるとよい（任意）。
- ダッシュボード(F3)に「復習待ち件数」を表示。

**受け入れ基準**:
- [ ] 誤答した問題が後日「復習」に出てくる（due管理が効いている）。
- [ ] 正解を重ねると出題間隔が延びる。
- [ ] 復習はリロード・日付またぎでも履歴が保たれる（`gken-srs-v1`）。
- [ ] 複数モードの採点が同一SRSに反映される。

### F3. 学習ダッシュボード（弱点分析）

**目的**: 「次に何を勉強すべきか」を提示する。

**要件**:
- トップ `index.html` 上部、または専用 `dashboard/index.html` に集約。
- 表示項目:
  - レッスン完了率（完了数 / 87）。
  - 問題カバレッジ（着手済み問題 / 567）。
  - **カテゴリ別正答率**（`gken-stats-v1.byCat`）。弱点カテゴリを強調し、該当章・復習へ誘導。
  - 連続学習日数（ストリーク）。
  - 復習待ち件数（F2）。
  - 直近の模試スコア推移（F1の `gken-exams-v1`）。
  - 学習プラン(F8)の今日のノルマ。
- すべて `localStorage` 集計。データが無い初期状態でも破綻しない（空状態の表示を用意）。

**受け入れ基準**:
- [ ] カテゴリ別正答率が表示され、弱点が視覚的に分かる。
- [ ] 完了率・カバレッジ・ストリーク・復習件数が表示される。
- [ ] データ未蓄積時に空状態が正しく出る。

### F4. サイト内検索

**目的**: 用語・レッスン・問題を横断的に引けるようにする。

**要件**:
- 生成時に軽量な検索インデックス `content/search-index.json` を [`scripts/generate-site.js`](../scripts/generate-site.js) で出力（レッスン: タイトル/用語/本文抜粋/URL、用語集語、必要なら問題見出し）。
- 検索UI（ヘッダの検索ボックス or `search/index.html`）。入力で**クライアント側フィルタ**、種別（レッスン/用語/問題）でグルーピング表示、結果クリックで該当ページへ。
- 日本語の部分一致で動作すること。

**受け入れ基準**:
- [ ] レッスン・用語を横断検索でき、結果から該当ページへ飛べる。
- [ ] サーバ無し（生成済みJSON + クライアント処理）で動作する。

### F5. 問題のブックマーク/フラグ

**目的**: 「後で見直す」問題を貯めて復習する。

**要件**:
- 演習・模試・復習の各問題に**ブックマーク切替**（★）を付与。`gken-bookmarks-v1`。
- ブックマーク一覧から「ブックマークだけ演習」を開始できる。
- F1の「後で見直す」フラグ（セッション内）とは区別（こちらは永続）。

**受け入れ基準**:
- [ ] 問題に★を付け外しでき、リロード後も保持される。
- [ ] ブックマークした問題だけを出題できる。

### F6. 問題⇄レッスンの相互リンク

**目的**: 演習と教材を往復できるようにし、誤答からの学び直しを速くする。

**要件**:
- 問題の `cat`（第N章）→ 章スラッグの対応表を用意。
- **問題の解説欄から、該当章インデックス（できれば最も関連するレッスン）へのリンク**を表示。
  - 関連レッスン推定は、既存 [`matchingQuestions()`](../scripts/generate-site.js) の逆（問題文・選択肢に含まれる用語からレッスンを引く）を流用してよい。
- **レッスンページに「関連する練習問題」リンク**（該当カテゴリ/用語でフィルタした演習へ）を追加。

**受け入れ基準**:
- [ ] 問題解説から関連する章/レッスンへ遷移できる。
- [ ] レッスンから関連演習へ遷移できる。

### F7. 進捗のエクスポート/インポート

**目的**: `localStorage` は端末依存。端末移行・バックアップを可能にする。

**要件**:
- 設定/ダッシュボードに「エクスポート」（全 `gken-*` をまとめたJSONをダウンロード）と「インポート」（JSON読み込みで復元）。
- バージョン/スキーマを含め、壊れた入力は弾く。

**受け入れ基準**:
- [ ] エクスポートしたJSONを別環境でインポートし、進捗・SRS・ブックマークが復元される。
- [ ] 不正JSONを読み込んでも既存データを壊さない。

### F8. 試験日からの学習プラン

**目的**: 残り日数から日々のノルマを示し、計画的に進められるようにする。

**要件**:
- 受験日入力（`gken-plan-v1.examDate`）。
- 残り日数と現状（未完了レッスン数・復習待ち件数・未着手問題数）から、**今日やるべき量の目安**を算出してダッシュボードに表示（例: 「レッスン2 / 復習30問 / 新規20問」）。
- 試験日超過時の表示も用意。

**受け入れ基準**:
- [ ] 受験日を入れると残り日数と今日のノルマが表示される。
- [ ] 進捗に応じてノルマが変化する。

## 6. 推奨実装順

1. **共通基盤（2章）** — 問題ID付与 + `store.js`。これが無いと他が積み上がらない。
2. **F2 SRS** — 採点更新APIを先に作る（各モードがフックする土台）。
3. **F1 模試（100分）** — 体感価値が最大。SRS連動込みで。
4. **F3 ダッシュボード** — F1/F2のデータを可視化。
5. **F5 ブックマーク** → **F6 相互リンク** → **F4 検索** → **F8 学習プラン** → **F7 エクスポート/インポート**。

## 7. 注意点

- **100分** は本番（web）の制限時間。模試タイマーの既定値はこれに合わせる。
- 模試の**問題数は1箇所の定数**にし、公式仕様変更に追従しやすくする（初期値の目安160、要確認）。
- 合否表示は必ず「目安」と明記し、公式基準と誤認させない。
- 解説HTML(`e`)は既存の `sanitizeExplanation()` 同様、`<b>` 以外のタグ・`on*`属性を除去して表示する。
- 既存 `practice.js` を作り直す場合も、現在の出題・採点UI（`is-correct`/`is-wrong`）は流用して体験を揃える。
- HTMLは生成物。ロジックは `assets/js/`、ページ骨組みは `generate-site.js` に置く。
